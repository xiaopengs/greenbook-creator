# Skill 设计规约（基于复旦-微软AI智能体技能评测研究）

> 来源论文：《From Raw Experience to Skill Consumption: A Systematic Study of Model-Generated Agent Skills》
> 团队：复旦大学、微软研究院、上海交通大学 | arXiv: 2605.23899 | 2026年5月
> 本规约基于论文实验验证的结论，非直觉推测。

---

## 一、核心设计原则：三维质量标准

论文通过150种实验组合（5领域 × 6消费者 × 5提炼者）的科学验证发现，以下三个维度能有效预测技能是否产生正向效果。**所有技能必须同时满足这三个维度。**

### 维度1：可操作的具体性（Actionable Specificity）

> 预测准确率：66.0%

**定义**：技能必须提供明确引用领域对象和工具的具体步骤，而非泛泛的指导。

| ❌ 错误示范 | ✅ 正确示范 |
|-----------|-----------|
| "要小心处理数据" | "使用 `.iloc` 进行基于位置的索引，避免列名匹配错误" |
| "注意线程安全" | "使用 `ConcurrentHashMap` 而非 `HashMap`，因为 SDK 的回调可能在多线程执行" |
| "确保异常处理" | "在 `chatStream()` 的 Flux 链中添加 `.onErrorResume()` 兜底，否则 SSE 连接会静默断开" |
| "处理好边界情况" | "当 `response.body()` 为 null 时，先检查 `response.isSuccessful()` 再读取，否则 Retrofit 会抛 NPE" |

**检查方法**：每条规则必须包含至少一个具体的类名、方法名、文件名或配置项。不含具体引用的规则视为无效。

### 维度2：失败机制编码（Failure Mechanism Encoding）

> 预测准确率：65.5%

**定义**：技能必须清晰说明失败的具体机制——不是泛泛地说"会出错"，而是解释什么条件触发、什么原因导致、什么症状表现。

| ❌ 错误示范 | ✅ 正确示范 |
|-----------|-----------|
| "这步可能出错" | "当 A 列有空值时，直接计算会触发 `TypeError`，需要先用 `fillna(0)` 预处理" |
| "注意超时问题" | "OkHttpClient 默认连接超时 10s、读取超时 10s。当模型回复超过 10s 时会抛 `SocketTimeoutException`，导致流式响应中断，用户只看到半截回复" |
| "数据库操作可能失败" | "在高并发写入时，PostgreSQL 的 unique 约束会触发 `PSQLException: duplicate key value`，需要用 `ON CONFLICT DO UPDATE` 替代简单的 INSERT" |

**检查方法**：每条关键规则必须回答三个子问题：
1. **触发条件**：在什么情况下会失败？
2. **根因**：为什么会失败（技术原因）？
3. **症状**：失败时用户/系统看到什么现象？

### 维度3：高风险操作黑名单（High-Risk Operation Blacklist）

> 预测准确率：64.6%

**定义**：技能必须明确列出绝对禁止的操作模式，并解释危害。

| ❌ 错误示范 | ✅ 正确示范 |
|-----------|-----------|
| "谨慎操作" | "绝对不要使用 `dropna()` 删除整行，这会破坏数据对齐" |
| "注意安全" | "禁止在日志中打印完整的 API Key 或 Token，即使是 DEBUG 级别。密钥泄露无法通过日志轮转撤回" |
| "不要阻塞主线程" | "禁止在 Android 主线程调用 `sessionManager.createSession().await()`，会导致 ANR。必须使用 `viewModelScope.launch` + `withContext(Dispatchers.IO)`" |

**检查方法**：每个技能必须有一个明确的"禁止项"列表，每项包含：
1. **禁止的操作**：具体什么操作不能做
2. **危害解释**：如果做了会发生什么
3. **替代方案**：应该怎么做

## 二、结构规范：Skill文件必须包含的模块

基于论文发现，"全面罗列型"（10+条详细规则）比"简洁聚焦型"（3-5条核心规则）效果差。因此：

### 2.1 必需模块

```
---
frontmatter 元数据（skill_id, name, role, group, mark, phase, priority）
---

# {skill-name} — {一句话描述}

## 目标
  （1-2句话说清楚这个技能要达成什么）

## 第一性原理
  （用一句话揭示本质矛盾，避免长篇大论）

## 高风险操作黑名单  ← ⭐ 新增必需模块
  （明确列出 DO NOT 操作 + 危害 + 替代方案）

## 编码规则
  （3-5 条核心规则，每条必须满足"可操作具体性"和"失败机制编码"）

## 失败机制速查  ← ⭐ 新增必需模块
  （列出本领域最常见的 3-5 种失败模式：触发条件 + 根因 + 症状 + 修复）

## SOP
  （步骤序列，每步必须引用具体的类名/方法名/API）

## 验证
  （检查清单）

## 产出物
  （交付物列表）
```

### 2.2 精简原则

- **编码规则不超过5条**：论文证明，3-5条核心规则 > 10+条全面规则
- **第一性原理不超过2句话**：过长会变成信息过载
- **SOP步骤不超过5个**：复杂流程应拆分为多个skill

## 三、质量检查清单

每个Skill在发布前必须通过以下审查：

### 3.1 三维质量检查

- [ ] **可操作的具体性**：每条编码规则是否引用了具体的类名/方法名/配置项？
- [ ] **失败机制编码**：是否解释了关键失败模式的触发条件、根因、症状？
- [ ] **高风险操作黑名单**：是否列出了明确的禁止操作 + 危害 + 替代方案？

### 3.2 信息负载检查

- [ ] 编码规则数量 ≤ 5条？
- [ ] 没有重复或高度重叠的规则？
- [ ] 没有显而易见的"废话"规则（如"写好代码"、"注意质量"）？

### 3.3 任务适配性检查

| 推荐度 | 任务类型 | 技能策略 |
|--------|---------|---------|
| ✅ 强烈推荐 | 规则明确、有确定性验证 | 技能详细、具体、含失败机制 |
| ⚠️ 谨慎使用 | 需要创造性探索 | 技能偏向原则性指导，不过度约束 |
| ❌ 不建议 | 无明确对错标准 | 不使用技能，依赖模型自身能力 |

## 四、元技能提示词

在 AI 消费技能时，在其系统提示词中加入以下"元技能"指导（论文验证平均提升 1.55 个百分点）：

```
在执行技能指导时，请遵循三个原则：
1. 严格按照具体的操作步骤执行，不要模糊理解或跳过任何具体引用；
2. 遇到失败机制描述的场景时，主动检查触发条件是否满足，而非等到出错才处理；
3. 将高风险操作黑名单视为硬约束，绝不尝试任何被禁止的操作。
```

## 五、持续监控机制

论文发现约25%的技能会产生负迁移（有技能反而表现更差）。因此：

1. **上线前 A/B 测试**：每个新技能必须对比"有技能"vs"无技能"的表现
2. **负迁移率监控**：超过 20% 的场景出现性能下降时，技能应下线或重写
3. **效果衰减跟踪**：定期检查技能是否因环境变化而过时
4. **失败案例反哺**：收集技能使用中的失败案例，用于更新失败机制编码

## 六、论文关键数据参考

| 指标 | 数据 | 含义 |
|------|------|------|
| 正向迁移率 | ~75% | 约四分之三的技能组合有正面效果 |
| 负向迁移率 | ~25% | 约四分之一的技能组合会帮倒忙 |
| 最佳提炼模式 | 简洁聚焦型 | 3-5条核心规则 > 10+条全面规则 |
| 元技能提升 | +1.55百分点 | 仅加入三维指导即可提升效果 |
| 表面标准效果 | -0.59百分点 | 使用未验证的标准反而会降低效果 |
